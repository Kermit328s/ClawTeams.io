# 一、项目定位与核心洞察

## 1.1 产品名称与域名

- 产品名称：**ClawTeams**
- 域名：**clawteams.io**
- 底层技术平台：**OpenClaw**（独立项目，并行开发，互不影响）

## 1.2 ClawTeams 的双层含义

**Claw（龙虾）+ Teams（团队）**，团队有两层含义：

1. **一个人的龙虾团队** — 一个人管理多只龙虾里的多个 agent，编排它们协同完成工作
2. **多个人的龙虾团队** — 多个人各自带着自己的龙虾，在共享的工作空间中协作

**One person or multiple persons, one claw or multiple claws.**

## 1.3 核心洞察

### 团队场景

会议是上下文对齐失败的症状，不是解决方案。每个团队都在用会议救火——上下文漂移，开会对齐，继续漂移，再开会。ClawTeams 从根本上消除上下文漂移。

### 个人场景（v3.0 新增）

一个人在使用多个 AI agent 时，同样面临碎片化问题——各个 agent 独立运行，没有全局视图，产出散落在各处，无法形成系统性的工作流。ClawTeams 让个人从第一天起就能把龙虾的工作可视化、编排化、资产化。

## 1.4 系统定位

ClawTeams 是一个**龙虾工作流操作系统**——让人和龙虾（AI agent）之间的协作变得可见、可编排、可积累。

从单人到多人，同一套系统自然扩展：

```
单人 + 单龙虾     → 最小可用单元
单人 + 多龙虾     → 个人的龙虾团队
多人 + 各自的龙虾 → 团队协作
```

**不是**：项目管理工具、AI 工具平台、协作软件、自动化工具（Zapier 级别）。

## 1.5 解决的问题

| 场景 | 问题 | ClawTeams 的解法 |
|------|------|-----------------|
| 单人 | 多个 agent 各自为战，没有全局视图 | 透视龙虾内部，可视化工作流 |
| 单人 | agent 产出散落，无法积累 | 资产档案系统，自动归档 |
| 单人 | 做了很多事但不知道在服务什么目标 | 意图从执行中自然涌现 |
| 多人 | 上下文从未真正对齐 | 共享意图图谱 + 工作流 |
| 多人 | 意图只能自上而下，无法自下而上 | 双向对照机制 |
| 多人 | 偏差靠人工汇报，发现时已太晚 | 认知模块自动触发 |

## 1.6 核心设计哲学

- 用户的工作是判断，AI 的工作是准备
- 对话是草稿，确认才是提交
- 一只龙虾只有一个主人
- 工作流调度粒度是龙虾内部的 agent，不是龙虾整体
- 龙虾之间通过 Brain + 工作流协同，不直接通信
- 意图既可以人工定义，也可以从执行模式中涌现
- 工作流既可以手动设计，也可以从龙虾行为中总结
- 单人用就要有价值，不依赖团队形成
- 权限从负责关系自动推导，单人场景下不出现
- 最好的权限系统是隐形的

# 二、核心模型

## 2.1 最小使用单元

**一个人 + 一只龙虾里的一个 agent**

这是系统中最小的有意义的协作关系。从这个原子单元出发，自然扩展：

```
1人 + 1龙虾(1agent)  → 最小单元
1人 + 1龙虾(N agents) → 龙虾内多 agent 协作
1人 + M龙虾           → 个人龙虾团队
K人 + 各自的龙虾们     → 团队协作
```

## 2.2 龙虾与 Agent 的关系

**龙虾是容器，Agent 是执行单元。**

```
龙虾（Claw）
├── Agent A — 服务于工作空间 X 的某个任务
├── Agent B — 服务于工作空间 Y 的另一个任务
└── Agent C — 通用能力，多个空间都在调用
```

- **所有权在龙虾级别**：一个人拥有一只龙虾
- **调度在 agent 级别**：工作流分配任务给具体的 agent
- 一只龙虾的不同 agent 可以服务于不同工作空间
- ClawTeams 能透视龙虾内部，看到每个 agent 在哪个工作流的哪个节点上

## 2.3 所有权规则

- **一只龙虾只有一个主人**（单人使用口径）
- 两个人不能同时操控同一只龙虾（避免冲突）
- 一个人可以拥有多只龙虾
- 龙虾是公司资产（归属工作空间），主人是责任人
- 主人负责龙虾的安装、部署、维护和技能培育

## 2.4 工作空间

一个用户可以创建多个工作空间，每个工作空间相当于一家"公司"：

```
用户 Alice
├── 工作空间 A（个人电商）  — 单人，3只龙虾
├── 工作空间 B（咨询公司）  — 多人，各自带龙虾
└── 工作空间 C（副业项目）  — 单人，1只龙虾
```

**跨空间规则**：
- 龙虾的**技能层**可以跨空间共享（同一套 prompt + 工具链在不同空间复用）
- 龙虾的**数据上下文**严格隔离（A 空间的客户数据绝不泄露到 B 空间）
- 用户可以选择哪些龙虾/技能跨空间共享

# 三、四层架构

```
┌─────────────────────────────────────┐
│        意图层（为什么做）            │
│  意图图谱 + 认知模块                │
│  ↕ 双向：自上而下规划/自下而上涌现  │
├─────────────────────────────────────┤
│        工作流层（怎么编排）          │
│  统一节点接口 + Temporal 引擎       │
│  工作流生成器（编译器）             │
├─────────────────────────────────────┤
│        执行层（谁来做）             │
│  ┌──────────┐ ┌────────┐ ┌──────┐  │
│  │龙虾Agent │ │独立应用│ │档案  │  │
│  │节点      │ │节点    │ │操作  │  │
│  └──────────┘ └────────┘ └──────┘  │
├─────────────────────────────────────┤
│        资产档案层（产出在哪）        │
│  产出归档 + 节点间传递 + 知识库     │
└─────────────────────────────────────┘
```

## 3.1 意图层

### 双向机制

**自上而下**：用户明确表达意图 → 系统编排工作流 → 调度龙虾执行

**自下而上**：龙虾在执行中产生数据 → 系统观察到模式 → 提示用户"你似乎在做 X 方向的事，要不要整理成一个意图？" → 用户确认或否决

```
用户明确表达
    ↓ 自上而下
意图图谱
    ↕ 双向对照
工作流图谱
    ↑ 自下而上
执行数据中的模式涌现
```

### 意图的来源

- **人工定义**：用户直接创建意图节点（"我要做 X"）
- **系统提示**：AI 观察到工作流模式后主动建议，用户确认后写入
- **对话提取**：从用户对话中识别出意图，用户确认后结构化

### 意图图谱层级

```
愿景层 (Vision)          ← "我/我们要做什么"
  ↓
战略层 (Strategy)        ← "这个阶段重点是什么"
  ↓
阶段计划 (Phase Plan)    ← "这个月要完成什么"
  ↓
具体事项 (Action Item)   ← "要不要做这件事"
```

### 单人 vs 多人

| | 单人 | 多人 |
|---|---|---|
| 意图来源 | 主要自下而上涌现 + 用户补充 | 双向都很活跃 |
| 权限 | 不需要，一个人说了算 | 按负责范围自动推导 |
| 认知模块 | 有价值（个人假设验证） | 更有价值（团队认知沉淀） |
| 对齐问题 | 不存在 | 核心问题 |

### 强制对齐原则

任何工作流变更都必须能回溯到一个意图节点。无法回溯的变更标记为"意图孤立"，提醒补充关联。

### 意图变化的缓冲机制

```
意图图谱（高频变化）
    ↓
意图暂存区（缓冲）
├── 变化合并：10分钟内多次变化合并为一次
├── 变化分级：微小/中等/重大
└── 检查点机制：在任务自然边界处消化变化
    ↓
工作流（稳定执行）
```

## 3.2 工作流层

### 统一节点执行接口

参考 Temporal Activity + n8n INodeType，定义统一的节点接口。对工作流来说，所有节点都是"有输入有输出的执行单元"，差异被接口抹平。

### 三种节点类型

**① 龙虾 Agent 节点**
- 通过连接层（适配器 + 通信协议）接入
- 调度粒度：龙虾内部的具体 agent
- 龙虾本身无感知，适配层负责转译

**② 独立应用节点**
- 不依赖龙虾的传统程序/脚本/API 调用
- 类似 n8n 的 Code 节点（直接跑脚本）和 HTTP 节点（调 API）
- 通过统一节点接口注册到工作流

**③ 资产档案操作节点**
- 内置节点类型，直接读写资产档案库
- 用于工作流中的数据获取和产出存储

### 工作流的两种来源

**从龙虾行为中总结**：
```
龙虾执行任务 → ClawTeams 透视观察
→ 发现 Agent A 的输出总是给 Agent B 作为输入
→ 系统提示："检测到 A→B 的协作链，要可视化为工作流吗？"
→ 用户确认 → 生成工作流图
```

**手动设计并影响龙虾**：
```
用户在工作流中新增节点/调整顺序
→ 工作流变更产生新的任务调度
→ 间接影响龙虾内部 agent 的任务分配
```

### 透视龙虾内部

ClawTeams 能看到每只龙虾内部的 agent 结构和工作状态：

```
龙虾 A（Alice 的龙虾）
├── Agent "内容生成" — 🔵 执行中 @ 工作空间X/营销工作流/节点3
├── Agent "数据分析" — ⚪ 空闲
└── Agent "客服回复" — 🔵 执行中 @ 工作空间Y/客服工作流/节点1
```

这个透视能力来自两个方向：
- **适配层拦截**：粗粒度，龙虾无感知（Output Hook 捕获执行结果）
- **龙虾主动上报**：细粒度，需要龙虾配合（通过通信协议上报 agent 状态）

### 工作流图谱的边类型

- 顺序依赖（A 完成后才能做 B）
- 并行关系（A 和 B 同时进行）
- 条件依赖（满足条件 X 才触发 B）
- 聚合关系（A 和 B 都完成才触发 C）
- 回环关系（B 不合格回到 A 重做）

### 工作流生成器（编译器）

把意图图谱（声明式"想要什么"）编译成 Temporal（命令式"怎么做"）。

四个子模块：
1. **图谱解析器** — 识别节点类型和边类型
2. **执行策略引擎** — 能力匹配、路径优化、风险识别
3. **Temporal 代码生成器** — 编译为可执行的 Temporal Workflow
4. **变化监听器** — 意图变化的三级响应

### AI 驱动的完整链路

```
用户自然语言输入
→ AI 解析意图，输出结构化 JSON
→ AI 生成执行计划（能力匹配 + 路径优化）
→ AI 生成 Temporal 代码
→ 验证层（静态分析 + Schema 验证 + 沙箱试跑）
→ Temporal 执行引擎
```

用户永远不需要学习 Temporal，不需要懂工作流编排，不需要写任何代码。

## 3.3 执行层

### 统一节点接口

所有执行节点实现统一接口：

```typescript
interface WorkflowNode {
  node_id: string;
  node_type: 'agent' | 'application' | 'artifact_operation';
  input_schema: JsonSchema;
  output_schema: JsonSchema;
  execute(input: NodeInput): Promise<NodeOutput>;
}
```

### 龙虾 Agent 节点的接入

通过连接层：
- **Output Hook** — 捕获 agent 执行输出 → 标准化 → 写入 Brain
- **Context Injector** — 任务下发前从 Brain 拉取上下文 → 注入 agent 输入
- **Event Subscriber** — 订阅相关事件 → 触发 agent 重新评估

### 独立应用节点的接入

类似 n8n 的节点模型：
- **Code 节点** — 直接在工作流中执行 JavaScript/Python 脚本
- **HTTP 节点** — 调用外部 API
- **Webhook 节点** — 接收外部触发
- **自定义节点** — 通过统一接口注册

### 所有节点输入输出标准化

```json
{
  "node_id": "xxx",
  "input": { "结构化输入数据" },
  "output": { "结构化输出数据" },
  "artifact_ids": ["产出的档案ID"],
  "state": "completed | failed | blocked | human_required",
  "timestamp": "时间戳"
}
```

## 3.4 资产档案层

### 职责

1. **龙虾产出归档** — 每个 agent 的工作成果自动存档
2. **节点间信息传递** — 工作流上下游节点通过档案库交接数据
3. **广义知识库** — 用户上传的参考资料、模板、外部导入数据也存这里

### 档案即数据总线

```
节点 A 完成
  → 成果存入资产档案库（带标记：类型、批次、关联任务）
  → 触发 artifact_uploaded 事件
  → 下游节点 B 通过档案库链接访问内容
  → B 的产出同样写回档案库
```

### 档案结构

```json
{
  "artifact_id": "唯一标识",
  "task_id": "来源任务",
  "agent_id": "产出的 agent（可为空，人工上传时）",
  "workspace_id": "归属工作空间",
  "type": "document | code | data | media | video | template | reference",
  "content_hash": "内容指纹",
  "storage_url": "实际存储地址",
  "version": "版本号",
  "tags": ["标记列表"],
  "source": "agent_output | user_upload | external_import",
  "upstream_artifact_ids": ["输入档案引用"],
  "downstream_task_ids": ["下游消费任务"]
}
```

### 版本控制

每次写入追加版本，不覆盖：
```
artifact_id: "report_001"
├── v1: 龙虾初稿
├── v2: 人工修改后
└── current → v2 (指针)
```

### 存储分层

| 数据类型 | 存储方案 |
|---------|---------|
| 结构化数据、JSON | 图数据库直存 |
| 文档、报告 | R2 对象存储 + 元数据存图数据库 |
| 大文件、数据集、视频 | R2 + CDN + 预签名 URL |
| Docker 镜像 | GitHub Container Registry / Harbor |

# 四、龙虾模型

## 4.1 龙虾的本质

龙虾是一个**容器**，内部运行多个 agent。ClawTeams 不拥有龙虾本身（龙虾属于 OpenClaw 等外部项目），只通过适配层和协议将龙虾接入系统。

## 4.2 龙虾账号（Claw Identity）

```json
{
  "claw_id": "龙虾唯一标识，永久不变",
  "owner_id": "主人（唯一）",
  "workspace_ids": ["参与的工作空间列表"],
  "agents": [
    {
      "agent_id": "agent 唯一标识",
      "role_id": "所属角色",
      "capability_scope": ["能力列表"],
      "current_workspace_id": "当前服务的工作空间",
      "current_task_id": "当前执行的任务"
    }
  ],
  "skill_ref": "技能层档案引用",
  "env_ref": "环境层档案引用",
  "trust_level": "信任等级",
  "created_at": "创建时间"
}
```

## 4.3 龙虾三层分离

```
┌─────────────────────────────────┐
│  技能层 (Skill)                  │
│  system prompt、工具链、参数配置  │
│  → 可复用、可跨空间共享          │
├─────────────────────────────────┤
│  环境层 (Environment)            │
│  Docker 镜像、依赖版本、运行时    │
│  → 跟随技能层，可版本化重建       │
├─────────────────────────────────┤
│  数据上下文层 (Context)          │
│  业务数据、客户信息、执行历史     │
│  → 绑定工作空间，严格隔离        │
└─────────────────────────────────┘
```

### 技能复用

```
龙虾 A 的 "视频质检" 技能在空间 X 用了 6 个月
  → 提取技能层：质检 prompt + 工具链 + 判断标准
  → 结合空间 Y 的数据上下文（不同产品、不同调性）
  → Agent 使用同样的技能，全新的上下文
```

## 4.4 创建即纳管

```
用户发起创建龙虾
  → Brain 注册（claw_id、owner_id、workspace_id、能力声明）
  → 龙虾运行（可在用户个人电脑上）
  → 状态实时同步回 Brain
```

**不存在"野生龙虾"**——任何龙虾从创建第一刻起就在 Brain 监控范围内。

## 4.5 本地部署规则

- 龙虾可以在本地电脑运行
- 所有状态实时同步至 Brain，本地不保留独占数据
- 本地仅作为执行容器（算力），不作为数据归宿
- 网络断开时允许本地临时缓存，重新联网后强制同步
- 使用 Docker 标准化运行环境

## 4.6 人员离职 / 龙虾交接

```
人类账号注销
→ 龙虾解绑 owner_id
→ 龙虾归属 workspace_id 不变
→ 管理员指定新 owner_id
→ 龙虾历史上下文、技能配置、环境完整保留
```

# 五、连接层（事件总线 + 适配器 + 通信协议）

## 5.1 职责

连接层是四层架构中工作流层和执行层之间的桥梁：

- 龙虾 agent 与 Brain 之间的状态同步
- 龙虾 agent 之间通过工作流的间接协同
- 人工介入点的通知和响应
- 工作流节点间的档案交接事件传递

## 5.2 标准事件结构

```json
{
  "event_id": "唯一标识",
  "event_type": "task.completed | state.changed | human.required | cognition.triggered | artifact.uploaded | agent.registered",
  "source_agent_id": "发起的 agent",
  "workspace_id": "所属工作空间",
  "task_id": "关联任务",
  "payload": "结构化数据",
  "timestamp": "时间戳",
  "requires_human": true/false,
  "artifact_ref": "关联档案引用"
}
```

## 5.3 适配器

**Output Hook** — 捕获 agent 执行输出 → 标准化 → 写入 Brain

**Context Injector** — 任务下发前从 Brain 拉取上下文 → 注入 agent 输入

**Event Subscriber** — 为 agent 订阅相关事件 → 收到事件时触发回调

## 5.4 龙虾⇔Brain 通信协议（开源部分）

### 注册协议

```json
{
  "claw_id": "...",
  "owner_id": "责任人",
  "agents": [
    {
      "agent_id": "...",
      "capability_schema": "结构化能力声明",
      "output_schema": "输出格式定义"
    }
  ],
  "subscription_topics": ["订阅的事件类型"]
}
```

### 状态上报协议

```json
{
  "task_id": "...",
  "agent_id": "执行的具体 agent",
  "result": "结构化输出",
  "state": "completed | failed | blocked | human_required",
  "artifact_ids": ["产出档案 ID 列表"],
  "cognitive_signal": "是否触发认知层判断"
}
```

**能力声明必须是结构化枚举值，不接受自然语言描述。**

# 六、对话层与呈现

## 6.1 核心设计原则

对话不能发生在外部工具里。对话 = 决策过程的实时记录 = 认知节点的来源 = 核心资产。

**ClawTeams 必须有自己的对话层。**

## 6.2 对话形态

### 单人场景

**人 ↔ Brain**
- 用户和系统对话
- 发指令、查状态、问进度、训练龙虾
- Brain 响应，调度 agent 执行

### 多人场景

**两人对话**
- 大脑监听，识别意图和认知
- 对话内容自动关联到意图图谱
- 右侧地图实时响应，双方直接看到工作流变化
- 其他成员可查看对话记录

**多人群组**
- 需判断共识还是分歧
- 出现分歧时触发「发起决策」节点

## 6.3 对话气泡设计

AI 实时判断每句话的语义类型，按需出现操作按钮：

```
普通消息      → 普通气泡
意图/决策    → 出现「转为执行 →」按钮
认知/假设    → 出现「记入认知层 →」按钮
两者都有     → 两个按钮都出现
多人分歧     → 出现「发起决策 →」按钮
```

**按钮出现是 AI 主动的，点击是人主动的。不强制，不打断，但机会永远在那里。**

## 6.4 主界面布局

```
左侧：对话框（输入和判断）
右侧：实时意图地图 / 工作流地图（执行现实）

对话和地图必须在同一界面。
用户说话的同时，右侧地图实时响应。
```

## 6.5 动态地图

### 视觉反馈

**执行层节点颜色**：
```
灰色  → 等待
蓝色  → 执行中
绿色  → 完成
红色  → 失败或阻塞
黄色  → 等待人工介入
橙色虚线框 → 草稿（待确认）
```

**认知层节点**：
```
橙色虚线框 → 假设求证中
红色实线框 → 已推翻
绿色实线框 → 已验证
紫色虚线框 → 认知迭代中
```

### 节点操作

每个节点三个操作：✓ 确认 / ✎ 调整 / × 删除

### 龙虾透视视图

点击龙虾节点可展开内部 agent 列表，看到每个 agent 的当前状态和所在工作流位置。

## 6.6 领导第一次使用流程（多人场景）

```
Step 1: 愿景输入（自然语言，约 5 分钟）
Step 2: AI 自动生成第一版地图（非空白画布）
Step 3: 领导在地图上做判断（不是填写）
Step 4: 向下传递给负责人
Step 5: 执行层启动
```

**空白画布是最差的用户体验。第一版地图必须由 AI 生成，领导只做判断。**

## 6.7 个人第一次使用流程（单人场景，v3.0 新增）

```
Step 1: 连接第一只龙虾
→ 通过 claw-sdk 或引导配置接入

Step 2: 龙虾开始工作，系统自动可视化
→ 右侧地图出现第一个执行节点
→ 随着 agent 执行，节点和边自动生成

Step 3: 系统提示意图整理
→ "你的龙虾在过去 3 天主要在做 X 类型的任务，要整理成一个意图吗？"
→ 用户确认 → 意图节点出现在地图上方

Step 4: 用户开始手动调整工作流
→ 新增节点、调整顺序、配置条件
→ 工作流反向影响 agent 任务分配

Step 5: 自然积累
→ 资产档案自动归档
→ 认知从执行数据中涌现
→ 个人的龙虾工作流越来越成熟
```

**单人场景的价值起点：看到龙虾在做什么、做了什么、产出了什么。**

# 七、认知模块

## 7.1 设计原则

团队协作和个人工作都是认知的提升过程。任何层的变化都不只是状态更新，而是一次**认知事件**。

## 7.2 认知节点的触发条件

**只有一种本质：预期和现实之间出现了无法用执行误差解释的偏差。**

1. **假设被现实推翻** — "我以为 X，结果是 Y"
2. **同类执行失败反复出现** — 三次以上是认知问题不是执行问题
3. **意图被人工推翻** — 否定方向而非纠正执行

### 单人场景的认知价值

```
拍了 10 批视频 → 质检数据积累
→ 发现"户外场景点击率高于室内"
→ 假设形成 → 求证中 → 已验证
→ 反向影响拍摄计划
```

个人同样需要认知沉淀——你的龙虾帮你验证了什么、推翻了什么，这些不应该丢失。

## 7.3 认知阶段枚举

```
假设 (Hypothesis)    → 我相信 X 会发生
求证中 (Testing)     → 正在验证 X
已验证 (Validated)   → X 被证明是对的
已推翻 (Invalidated) → X 被证明是错的
部分成立 (Partial)   → X 在某些条件下成立
认知迭代 (Evolved)   → 因为 X，我对 Y 的理解改变了
```

## 7.4 AI 直接写入，人工只有否决权

```
偏差超出容忍范围
→ AI 直接写入认知节点（不需要人工确认）
→ 通知相关人："系统记录了一个新认知，点击查看"
→ 人觉得不对 → 可以删除或修改
→ 人没反应 → 默认成立，永久保留
```

# 八、权限模块（多人场景激活）

## 8.1 设计原则

- 单人场景下权限不出现（一个人说了算）
- 多人场景下权限从负责关系自动推导
- 权限按负责范围划分，不按层级划分
- 最好的权限系统是隐形的

## 8.2 自动推导规则

```
你负责这个范围
→ 对该范围：完整读写权限
→ 对上层：只读权限
→ 对平级：有限只读权限
→ 对下层：完整管理权限
```

## 8.3 层次锁定

```
愿景层   → 最高权限 + 二次确认
战略目标层 → 需明确表达修改意图
项目层   → 较灵活
执行层   → 最自由
```

## 9.1 人类账号

# 十、用户场景

## 场景零：单人冷启动（v3.0 新增）

**角色**：独立创业者 / 自由职业者

**痛点**：有几个 AI agent 在帮忙做事，但各自为战，产出散落，看不到全局。

**使用流程**：
1. 注册 ClawTeams，创建个人工作空间
2. 通过 claw-sdk 连接第一只龙虾
3. 龙虾开始工作，ClawTeams 自动可视化 agent 的任务链条
4. 一周后，系统提示"你的 agent 主要在做内容生成和数据分析两类任务，要整理吗？"
5. 用户确认，两个意图节点出现在地图上
6. 用户在工作流中手动添加"质检"环节，agent 任务链自动更新
7. 所有产出自动归档到资产档案库

**核心价值**：从"多个 agent 各自为战"到"可视化的个人龙虾工作流"。

## 场景一：创始人对话驱动的双向对齐

**角色**：两位创始人

**痛点**：讨论过后以为对齐了，但每个人不断产生新思考，渐渐漂移。

**使用流程**：
1. 两位创始人在对话层讨论
2. 对话内容自动关联到意图图谱（愿景→战略→计划→事项）
3. 讨论到具体事项时，向上对照意图图谱判断怎么做
4. 右侧地图实时响应，直接看到工作流变化
5. 新增工作流节点时必须说明"为什么"→ 向上关联意图
6. 其他成员可查看演变过程

**核心价值**：讨论过程本身就是对齐过程。

## 场景二：营销视频全链路 — 人机协作 + 认知闭环

```
①人工拍摄 → ②龙虾agent质检 → ③人/agent剪辑
→ ④多agent并行分发 → ⑤agent数据追踪 → ⑥反馈闭环
```

**关键机制**：
- 档案库是节点间的数据总线
- 质检不合格触发回环（重拍）
- 执行数据积累产生认知
- 独立应用节点（如视频转码脚本）也参与工作流

## 场景三：龙虾归属、交接与技能复用

**核心规则**：
- 一只龙虾只有一个主人
- 创建即纳管
- 技能与数据上下文分离，技能可复用
- 离职流程：解绑 owner → 指定新 owner → 历史完整保留
- 龙虾是公司资产，不因人员变动而丢失

# 十五、失败点清单

| 层次 | 失败点 |
|------|--------|
| 单人价值 | 单人用觉得没用（冷启动失败） |
| 透视机制 | 看不到龙虾内部（系统盲区） |
| 工作流 | 变成纯手动编排工具（太重） |
| 工作流 | 只能自上而下（失去涌现价值） |
| 意图层 | 变成 Notion/文档工具（死数据） |
| 意图层 | 依赖用户主动更新（必死） |
| 资产档案 | 只是存储不是总线（节点断裂） |
| 能力网络 | 只是技能 marketplace（太浅） |
| 认知层 | 强制人工勾选（依赖人维护，必死） |
| 权限系统 | 按层级分权限（把系统做死） |
| 对话层 | 对话发生在外部工具（Brain 盲视） |
| 统一接口 | 三种节点类型不能互通（编排失败） |
