# ClawTeams 四阶段开发总览

---

## 递进关系

```
阶段一：看见（读取）
  ClawTeams ← 文件 hash + Hook ← OpenClaw
  "能看到 Agent 是谁、在做什么、做了什么"

阶段二：回写（写入）
  ClawTeams → Gateway 客户端 → Agent 自己改 → hash 验证
  "能通过 ClawTeams 影响 Agent 的定义和行为"

阶段三：编排（完整控制）
  统一节点接口 + 意图层 + 对话层 + 认知
  "能设计工作流、整理意图、积累认知"

阶段四：协同（多人）
  多人工作空间 + 权限 + 团队认知
  "多人各自带龙虾在共享空间协作"
```

---

## 阶段一：看见（读取）

### 目标
一个人连上龙虾后，能看到龙虾在做什么、做了什么、产出了什么。

### 核心能力
- 文件 hash 追踪：读取 Agent 核心文件，建立画像
- 会话 JSONL 增量解析：提取任务历史
- 5 个轻量 Hook：实时状态（上线/执行/Subagent）
- 前端：Agent 面板 + 自动工作流图 + 资产列表 + 活动日志

### 数据流方向
```
OpenClaw 文件系统 ──→ ClawTeams（只读）
OpenClaw Hook ──→ ClawTeams（只读）
```

### 验证标准
- 接入一只龙虾，界面上看到 Agent 列表和状态
- 看到 Agent 核心文件内容和版本变化
- 看到任务历史和自动生成的工作流图

### 详细规划
见 `docs/phase1_开发规划.md`

## 阶段二：回写（写入）

### 目标
能通过 ClawTeams 的操作影响 Agent 的定义和行为，形成读写闭环。

### 核心能力
- Gateway WebSocket 客户端：模拟正常客户端接入 OpenClaw
- 指令生成：将 ClawTeams 工作流变更翻译为 Agent 可执行的自然语言指令
- 回写验证：文件 hash 追踪检测变更是否生效

### 数据流方向
```
ClawTeams ──→ Gateway WS 客户端 ──→ Agent 执行 ──→ Agent 修改自己的文件
                                                         ↓
ClawTeams ←── 文件 hash 追踪检测变更 ←───────────────────┘
```

### 回写闭环

```
1. ClawTeams 产生工作流变更
2. 翻译为自然语言指令
3. 通过 Gateway WS 发送给对应 Agent
4. Agent 理解指令，执行修改（用 exec 等工具）
5. 文件 hash 追踪检测到变更
6. 对比 diff 确认改的对不对
7. 成功 → 确认回写，更新版本
   失败 → 超时未检测到变更 → 标记失败，提示用户
```

### 回写场景清单

| ClawTeams 操作 | 生成的指令 | 修改的文件 |
|---------------|----------|----------|
| 新增 Agent | "创建一个新的 Agent，角色是 X，负责 Y" | 新建 SOUL.md + IDENTITY.md + TOOLS.md |
| 修改 Agent 角色 | "更新你的身份定义为 X" | IDENTITY.md |
| 调整 Agent 行为边界 | "在你的 SOUL 中添加/修改规则 X" | SOUL.md |
| 修改协作关系 | "更新 AGENTS.md，添加与 Agent B 的协作" | AGENTS.md |
| 调整定时任务 | "在 HEARTBEAT.md 中添加每天 9 点执行 X" | HEARTBEAT.md |
| 调整工具权限 | "更新 TOOLS.md，允许/禁止使用工具 X" | TOOLS.md |
| 写入认知/记忆 | "记住这个结论：X" | memory |
| 调整模型 | "切换你的模型为 X" | openclaw.json |

### 关键技术点

**Gateway 客户端实现**：
- 通过 OpenClaw Gateway WS 协议接入（端口 18789）
- 使用设备配对认证（与 Web UI 相同的接入方式）
- 以 "clawteams" Channel 身份发送消息
- 支持接收 Agent 的流式响应

**指令生成引擎**：
- 将 ClawTeams UI 操作转换为结构化的 Agent 指令
- 指令模板库：每种操作对应一个指令模板
- 指令中包含明确的文件路径和预期修改内容
- 支持复合操作（一个工作流变更可能需要多条指令）

**回写验证**：
- 发出指令后启动 hash 监控
- 超时时间：30 秒（Agent 需要时间执行）
- 验证内容：不仅检测文件变了，还要对比 diff 确认改的方向正确
- 失败重试：最多 2 次，每次调整指令措辞

### 验证标准
- 在 ClawTeams 前端修改一个 Agent 的角色 → IDENTITY.md 自动更新
- 在 ClawTeams 前端添加定时任务 → HEARTBEAT.md 自动更新
- 在 ClawTeams 前端修改协作关系 → AGENTS.md 自动更新
- 修改失败时用户收到明确提示

### Sprint 规划

**Sprint 1: Gateway 客户端（1-2 周）**

| 任务 | 说明 |
|------|------|
| S1-1 | 研究 OpenClaw Gateway WS 协议和认证流程 |
| S1-2 | 实现 Gateway WS 客户端（设备配对认证） |
| S1-3 | 实现消息发送和响应接收 |
| S1-4 | 测试：通过客户端给 main Agent 发消息并收到回复 |

**Sprint 2: 指令生成 + 回写验证（1-2 周）**

| 任务 | 说明 |
|------|------|
| S2-1 | 设计指令模板库（每种操作 → 自然语言指令） |
| S2-2 | 实现回写验证：指令发出 → hash 监控 → diff 对比 |
| S2-3 | 实现失败检测和重试机制 |
| S2-4 | 测试：修改 IDENTITY.md、HEARTBEAT.md、AGENTS.md |

**Sprint 3: 前端集成（1-2 周）**

| 任务 | 说明 |
|------|------|
| S3-1 | Agent 核心文件编辑界面（展示内容 + 修改按钮） |
| S3-2 | 工作流图编辑 → 触发回写 |
| S3-3 | 回写状态指示（执行中/成功/失败） |
| S3-4 | 版本 diff 查看（修改前 vs 修改后） |

## 阶段三：编排（完整控制）

### 目标
能设计完整的工作流、整理意图、积累认知。

### 核心能力
- 统一节点接口：Agent 节点 + 独立应用节点 + 资产档案操作节点
- 工作流手动编辑：新增/删除/调整节点和边
- 工作流生成器基础版：手动编排 → Temporal 执行
- 意图层：自下而上涌现（从文件版本演化提示）+ 用户手动补充 + 双向对照
- 认知基础版：偏差检测 + 假设记录
- 对话层：人 ↔ Brain

### 建立在前两个阶段之上
```
阶段一提供：Agent 画像 + 运行时状态 + 任务历史（读）
阶段二提供：回写能力 + Gateway 客户端（写）
阶段三新增：编排逻辑 + 意图 + 认知 + 对话
```

### 关键新增功能

**统一节点接口**：
- Agent 节点：通过阶段二的 Gateway 客户端调度
- 独立应用节点：直接执行脚本/API
- 资产档案操作节点：内置读写

**意图从文件演化中涌现**：
```
SOUL.md 改了 3 次 → 行为边界在收敛
AGENTS.md 从 3 个协作变成 6 个 → 团队在扩展
HEARTBEAT.md 新增了定时扫描 → 自动化程度在提高
  ↓
系统提示："你的 Agent 体系在朝 X 方向演化，要整理成意图吗？"
```

### 验证标准
- 能设计一条多节点工作流（混合 Agent + 脚本 + 档案）并执行
- 系统能从文件版本演化中提示意图整理
- 对话中的决策能关联到意图图谱

## 阶段四：协同（多人）

### 目标
多个人各自带龙虾在共享工作空间中协作，上下文保持对齐。

### 核心能力
- 多人工作空间：邀请、成员管理
- 对话层完整版：两人对话、群组、智能气泡
- 权限模块：按负责范围自动推导
- 认知完整版：团队认知沉淀
- 工作流编译器完整版：AI 编译器、变化监听器
- 跨空间技能共享 + 数据隔离

### 建立在前三个阶段之上
```
阶段一：每个人的龙虾状态可见
阶段二：每个人能通过 ClawTeams 控制自己的龙虾
阶段三：每个人能编排自己的工作流和意图
阶段四新增：多个人的龙虾在同一工作空间协同
```

### 验证标准
- 两个人在同一工作空间，各自龙虾的 Agent 在工作流中协同
- 对话中实时看到地图变化
- 权限自动推导，角色负责范围内可自由操作

## 四阶段时间线预估

```
阶段一：看见     5 Sprint  ≈ 6-10 周
阶段二：回写     3 Sprint  ≈ 4-6 周
阶段三：编排     4 Sprint  ≈ 6-8 周
阶段四：协同     4 Sprint  ≈ 6-8 周
                         ──────────
                  总计 ≈ 22-32 周
```

阶段一完成即可面向单人用户发布 MVP。
阶段二完成后用户可以通过 ClawTeams 完整管理龙虾。
阶段三完成后产品核心价值闭环。
阶段四完成后具备团队协作能力。
