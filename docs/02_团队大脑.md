## 1. 板块职责

团队大脑是 ClawTeams 的核心，负责维护团队的共享上下文——意图图谱、认知沉淀、权限体系和账号管理。所有其他模块围绕大脑运转。

## 2. 账号模块

### 人类账号（User Account）

- 标准 JWT 登录体系
- 有会话过期，人不在线即失效
- 权限归属：个人权限

### 龙虾账号（Agent Identity）

```json
{
  "agent_id": "唯一标识，永久不变",
  "role_id": "所属角色（非人名）",
  "owner_id": "绑定用户，可为空",
  "team_id": "所属团队",
  "capability_scope": ["可调用的能力列表"],
  "trust_level": "信任等级",
  "created_at": "创建时间",
  "state_snapshot_ref": "最新状态快照引用",
  "skill_ref": "技能层档案引用",
  "env_ref": "环境层档案引用",
  "context_domain": "数据上下文所属域"
}
```

- 长期有效 API Key 机制
- 7×24 持续存在，不依赖人类登录态
- 权限归属：角色权限

### 所有权模型

- 一个用户可拥有多个龙虾
- Owner 对龙虾工作效果负责
- 龙虾是公司资产，不是个人资产（归属 team_id）
- 可授权龙虾代表自己行动
- 可设定龙虾的能力边界
- 可随时撤销授权

### 创建即纳管

```
用户发起创建龙虾
  → 团队大脑注册（agent_id、owner_id、team_id、能力声明）
  → 龙虾运行（可在用户个人电脑上）
  → 状态实时同步回团队大脑
```

**不存在"野生龙虾"**——任何龙虾从创建第一刻起就在大脑监控范围内。

### 人员离职流程

```
人类账号注销
→ 龙虾解绑 owner_id
→ 龙虾归属 team_id 不变
→ 团队管理员指定新 owner_id
→ 新人登录，重新授权
→ 龙虾历史上下文、技能配置、环境完整保留
```

## 3. 意图图谱（Intent Graph）

### 层级结构

```
愿景层 (Vision)          ← "公司到底要做什么"
  ↓
战略层 (Strategy)        ← "这个阶段重点是什么"
  ↓
阶段计划 (Phase Plan)    ← "Q2要完成哪几件事"
  ↓
具体事项 (Action Item)   ← "要不要做这件事"
```

### 四个子模块

**Intent Graph（意图图）**
- 当前激活目标（active goals）
- 子任务分解树（task tree）
- 优先级与依赖关系

**State Timeline（状态时间线）**
- 谁做了什么（actions log）
- 当前进度（progress states）
- 变化差异记录（diff log）

**Knowledge Memory（知识沉淀）**
- 已验证信息、决策依据、历史结论与经验

**Social Context（关系上下文）**
- 角色与负责范围、信任权重与authority、协作关系图谱

### 与工作流的双向对照机制

意图图谱和工作流之间不是单向的"意图→执行"，而是**双向实时对照**：

**向下（意图→工作流）：**
- 讨论一个战略方向 → 自动影响下游工作流的节点安排
- 工作流图谱实时产生变化（新增节点/分叉/优先级调整）

**向上（工作流→意图）：**
- 要做一件具体事时 → 返回意图图谱对照 → 这件事服务于哪个战略目标？
- 新增工作流节点必须回答"对应哪个意图"
- 局部意图与整体图谱产生一致性检阅

```
意图图谱（为什么做）
   ↕ 双向对照
工作流图谱（做什么、怎么做）
   ↕
节点操作（增加/分叉/调整）必须说明"为什么"
→ "为什么"自动向上关联到意图图谱
→ 局部意图 vs 整体图谱产生一致性检阅
```

**强制对齐原则**：任何工作流变更都必须能回溯到一个意图节点。无法回溯的变更系统标记为"意图孤立"，提醒补充关联。

### 状态写入规则

- 自动写入，不依赖人工维护
- 强语义结构，不接受纯文本
- 版本化存储，支持回滚
- 事件驱动，支持订阅

### 回滚机制

**单龙虾回滚**：大脑下发指令 → 龙虾恢复指定时间点状态 → 重新执行

**团队整体回滚**：大脑协调所有相关龙虾 → 同步回滚到同一时间点 → 保持状态一致性

## 4. 权限模块

### 设计原则

- 权限按**负责范围**划分，不按层级划分
- 负责某个范围的人，对该范围的所有层次都有完整参与权
- 权限从负责关系自动推导，无需手动配置
- 最好的权限系统是隐形的

### 自动推导规则

```
你负责这个范围
→ 对该范围：完整读写权限（意图到执行所有层次）
→ 对上层：只读权限
→ 对平级：有限只读权限
→ 对下层：完整管理权限
```

### 层次锁定机制

```
愿景层   → 最高权限 + 二次确认才能修改
战略目标层 → 需明确表达修改意图才触发
项目层   → 较灵活，影响下游时需确认
执行层   → 最自由，随时可调整
```

### 三种例外需人工配置

1. 跨范围协作
2. 临时授权
3. 敏感内容保护

### 任务节点的权限参数

```python
class TaskInput:
    required_capability: str   # 需要什么龙虾能力
    agent_id: str              # 指定哪个龙虾
    permission_level: str      # 对应Brain权限层级
    approver_role: str         # 需要什么角色审批
    timeout_hours: int         # 等待人工响应超时
    auto_approve: bool         # 超时是否自动通过
    escalation_user_id: str    # 超时升级给谁
```

## 5. 认知模块

### 设计原则

团队协作本身是集体认知的提升过程。任何层的变化都不只是状态更新，而是一次**认知事件**。

### 认知节点与执行节点的关系

```
执行节点 → 描述要做什么
认知节点 → 描述为什么这么做、学到了什么

假设节点 → 驱动 → 执行节点
执行节点 → 产生 → 认知节点
认知节点 → 修正 → 假设节点
```

### 触发条件

**只有一种本质：预期和现实之间出现了无法用执行误差解释的偏差。**

1. **假设被现实推翻** — "我们以为世界是X，结果世界是Y"
2. **同类执行失败反复出现** — 单次失败是意外，三次以上是认知问题
3. **意图被人工推翻** — 否定方向而非纠正执行错误

### 认知从执行中自然涌现

```
偏差在容忍范围内 → 状态机记录，跳过认知层
偏差超出容忍范围 → AI直接写入认知节点（不需要人工确认）
    → 通知相关人："系统记录了一个新认知，点击查看"
    → 人觉得不对 → 可以删除或修改
    → 人没反应 → 默认成立，永久保留
```

**关键原则：AI直接写入，人工只有否决权。**

### 认知阶段枚举

```
假设 (Hypothesis)    → 我们相信X会发生
求证中 (Testing)     → 正在验证X
已验证 (Validated)   → X被证明是对的
已推翻 (Invalidated) → X被证明是错的
部分成立 (Partial)   → X在某些条件下成立
认知迭代 (Evolved)   → 因为X，我们对Y的理解改变了
```
