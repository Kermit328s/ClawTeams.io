---

## 1. 板块职责

前端是用户与 ClawTeams 交互的唯一界面，包含对话层和动态地图两大核心。对话层是决策的入口，动态地图是执行现实的实时镜像。两者必须在同一界面，不是两个页面。

**第一层：对话框（输入和判断）** — 用户与系统/其他人的对话入口

**第二层：动态地图（执行现实的实时镜像）** — 意图图谱 + 工作流的可视化

## 4. 对话层

### 4.1 核心设计原则

对话不能发生在飞书、Slack、Telegram等外部工具里，因为：

```
对话 = 决策过程的实时记录
决策过程 = 认知节点的来源
认知节点 = 团队大脑的核心资产
```

外部工具让大脑系统对决策过程完全盲视。**ClawTeams必须有自己的对话层。**

### 4.2 三种对话形态

**① 人 ↔ 大脑**
- 用户和ClawTeams系统对话
- 发指令、查状态、问进度
- 大脑响应，龙虾执行
- 意图最清晰，直接执行

**② 两人对话**
- 两个人在讨论一件事
- 大脑监听，识别意图和认知
- 适时出现操作按钮，不打断对话节奏
- 需判断谁的意图算数
- 对话内容自动关联到意图图谱对应层级
- 讨论过程中右侧地图实时响应，双方直接看到工作流变化
- 其他成员可查看对话记录，看到决策是怎么演变来的

**③ 多人群组**
- 多人同时参与
- 需判断共识还是分歧
- 出现分歧时不随便选一方意见执行
- 触发「发起决策」节点，等有权限的人拍板后再执行

### 4.3 对话气泡的特殊设计

AI实时判断每句话的语义类型，按需出现操作按钮：

```
普通消息      → 普通气泡，不带任何按钮
意图/决策    → 出现「转为执行 →」按钮
认知/假设    → 出现「记入认知层 →」按钮
两者都有     → 两个按钮都出现
多人分歧     → 出现「发起决策 →」按钮
```

**按钮出现是AI主动的，点击是人主动的。不强制，不打断，但机会永远在那里。**

### 4.4 气泡状态卡片（点击转为执行后）

气泡原地变形为活的状态卡片，实时反映执行进度：

```
点击前：
  「我想关掉几个效果差的ad」
  [转为执行 →] [记入认知层 →]

↓ 点击「转为执行」

执行中：
┌──────────────────────┐
│ 关掉效果差的广告      │
│ ──────────────────── │
│ 🔵 龙虾拉取广告数据中…│
└──────────────────────┘

↓ 数据回来

待确认：
┌──────────────────────┐
│ ✓ 找到4个CTR低于0.8%  │
│ [查看名单] [确认关闭]  │
└──────────────────────┘

↓ 确认后

完成：
┌──────────────────────────┐
│ 关掉效果差的广告  ✓       │
│ ──────────────────────── │
│ 已关闭4个广告             │
│ 3天后自动回来验证CTR变化   │
│ [查看详情]                │
└──────────────────────────┘
```

**对话记录本身就是任务的完整时间轴。**

### 4.5 对话驱动意图图谱更新

来自场景一的核心机制：

**创始人对话中的双向对齐**

```
创始人A ←→ 创始人B 在对话中自然讨论
    ↓ 大脑实时监听
    ↓ 对话内容自动关联到意图图谱对应层级

"我们公司到底要做什么"  → 关联到愿景层
"这个阶段重点是什么"    → 关联到战略层
"Q2要完成哪几件事"      → 关联到阶段计划
"要不要做这件事"        → 关联到具体事项
```

**讨论即演变记录**：
- 对话过程被结构化保存
- 其他搭档/公司成员可以看到意图是怎么演变来的
- 不只看到结论，看到讨论过程中的思考碰撞

**实时地图响应是核心爽感**：
- 对话过程中右侧地图在动
- 能直观看到"我们刚讨论的这个方向会影响哪些项目"
- 变更先以草稿态呈现，确认后正式生效

## 5. 动态地图

### 5.1 视觉反馈层次

**生成层（AI在工作）**
- 节点从模糊到清晰
- 用户可以在此阶段打断和修正

**编排层（计划在形成）**
- 边和依赖关系动态连接
- 用户可以拖拽调整依赖关系

**执行层（龙虾在跑）**
```
灰色  → 等待
蓝色  → 执行中
绿色  → 完成
红色  → 失败或阻塞
黄色  → 等待人工介入
橙色虚线框 → 草稿（待确认）
```

**认知层（分隔线以下）**
```
橙色虚线框 → 假设求证中
红色实线框 → 已推翻
绿色实线框 → 已验证
紫色虚线框 → 认知迭代中
```

### 5.2 地图节点操作

每个节点三个操作：✓ 确认 / ✎ 调整 / × 删除

### 5.3 Temporal 状态实时映射

```
Temporal Activity 状态变化
→ 触发事件写入事件总线
→ 前端订阅事件流（WebSocket）
→ 实时更新地图节点状态
```

用户看到的不是 Temporal，是意图地图的实时演化。

### 5.4 任务路由状态可视化

```
纯龙虾节点  → 蓝色自动流转
人工节点     → 黄色高亮 + 推送通知
混合节点     → 蓝色执行完变黄色等待审批
权限不匹配  → 红色 + 说明需要谁处理
超时升级     → 节点闪烁 + 通知升级对象
```

## 6. 自然语言保护机制

```
自然语言输入
→ AI判断输入类型：
   ├── 新增内容      → 在现有结构上添加节点
   ├── 局部调整      → 只修改相关节点
   ├── 结构性变化    → 展示影响范围预览，需确认
   └── 查询/闲聊    → 不触发任何修改
→ 修改先进入草稿层
→ 用户确认后合并进正式结构
```

**影响范围预览：**
```
你的这个调整会影响：
- N个项目的时间线
- N个部门的任务分配
- N个已确认的里程碑
确认修改？ [是] [取消]
```

## 7. 领导第一次使用流程

```
Step 1: 愿景输入（自然语言，约5分钟）
→ 系统引导，领导自由表达
→ AI结构化，领导确认

Step 2: AI自动生成第一版地图（非空白画布）
→ 推导战略目标（3-5个）
→ 推导功能板块
→ 推导第一阶段时间线

Step 3: 领导在地图上做判断（不是填写）
→ 逐节点确认/调整/删除
→ AI实时更新地图

Step 4: 向下传递（自动）
→ 相关板块推送给负责人
→ 负责人看到AI建议的任务分解
→ 负责人确认/调整后继续向下传递

Step 5: 执行层启动
→ 龙虾开始接收任务
→ 适配层注入上下文
→ 执行结果实时回写
```

**空白画布是最差的用户体验。第一版地图必须由AI生成，领导只做判断。**
