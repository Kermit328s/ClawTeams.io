> 本文档定义代理分工、协同机制和顶层监督规则

## 2. 各代理职责

### 2.1 Architect（架构师代理）— 顶层监督

**不写代码，只做决策和审查。**

| 职责 | 说明 |
|------|------|
| 接口仲裁 | 当两个板块代理对接口定义有分歧时，Architect 拍板 |
| Schema 审查 | 所有数据模型变更必须经 Architect 审批 |
| 一致性检查 | 定期检查各板块的实现是否偏离需求文档 |
| 依赖协调 | 管理板块间的启动顺序和阻塞问题 |
| 变更广播 | 当一个板块的接口变更影响其他板块时，通知所有受影响方 |
| 集成验证 | 各板块交付后，验证端到端链路是否跑通 |

**决策权限：**
- 可以否决任何板块代理的接口变更
- 可以调整板块间的优先级
- 不可以绕过板块代理直接修改板块内部实现

### 2.2 Infra（基础设施代理）

**所有板块的前置依赖，最先启动。**

| 职责 | 产出物 |
|------|--------|
| 最小状态单元定义 | `proto/state_unit.proto` 或 JSON Schema |
| 数据库 Schema | Neo4j 节点/边类型定义、关系数据库表结构 |
| 存储配置 | R2/MinIO bucket 规划、预签名 URL 服务 |
| Docker 基础镜像 | 龙虾运行的标准基础镜像 |
| 共享类型定义 | 各板块共用的 TypeScript/Python 类型包 |
| API 网关 | 统一入口、鉴权中间件 |

**上游依赖**：无（第一个启动）
**下游消费者**：所有其他板块代理

### 2.3 Brain（大脑代理）

| 职责 | 产出物 |
|------|--------|
| 账号模块 | 人类账号 CRUD、龙虾账号 CRUD、绑定关系 |
| 意图图谱 | Intent Graph CRUD、四层节点、双向对照 API |
| 权限模块 | 自动推导引擎、层次锁定、例外配置 |
| 认知模块 | 触发判断、认知写入、阶段枚举、否决机制 |
| 回滚机制 | 单龙虾回滚、团队整体回滚 |

**上游依赖**：Infra（Neo4j Schema、共享类型）
**下游消费者**：Workflow（读取意图图谱）、Frontend（展示数据）、Connector（龙虾注册）

### 2.4 Workflow（工作流代理）

| 职责 | 产出物 |
|------|--------|
| 图谱解析器 | Graph Parser（顺序/并行/条件/回环） |
| Temporal 编译器 | 意图图谱 → Temporal Workflow 代码 |
| 执行策略引擎 | 能力匹配、路径优化、风险识别 |
| 变化监听器 | 意图缓冲、检查点、三级变化响应 |
| AI 链路 | 自然语言 → 结构化 JSON → 执行计划 → Temporal |

**上游依赖**：Infra（Temporal 环境）、Brain（意图图谱 API）
**下游消费者**：Connector（任务下发）、Frontend（工作流状态）

### 2.5 Connector（连接层代理）

| 职责 | 产出物 |
|------|--------|
| 事件总线 | Kafka/WebSocket 基础设施、事件路由 |
| 适配器 | Output Hook、Context Injector、Event Subscriber |
| 通信协议 | 注册/上报/订阅协议定义（开源） |
| 协议验证工具 | 测试龙虾是否符合协议标准 |
| 龙虾状态同步 | 三层信息同步、断网缓存、重连同步 |

**上游依赖**：Infra（事件结构定义）、Brain（龙虾账号 API）
**下游消费者**：Workflow（事件触发）、Frontend（WebSocket 推送）、外部龙虾（协议边界）

### 2.6 Frontend（前端代理）

| 职责 | 产出物 |
|------|--------|
| 对话层 | 三种对话形态、气泡设计、状态卡片 |
| 动态地图 | 节点可视化、四层视觉反馈、拖拽交互 |
| 实时同步 | WebSocket 订阅 → 地图节点实时更新 |
| 自然语言保护 | 输入分类、影响范围预览、草稿确认 |
| 引导流程 | 首次使用 5 步引导 |

**上游依赖**：Brain（API）、Connector（WebSocket 事件流）
**下游消费者**：用户

## 3. 协同机制

### 3.1 共享契约（Contract）

板块之间通过**接口契约**协同，不通过直接调用内部实现。

```
contracts/
├── brain-api.yaml          # Brain 对外暴露的 API 契约
├── workflow-api.yaml       # Workflow 对外暴露的 API 契约
├── event-schema.yaml       # 事件总线的事件结构契约
├── protocol-spec.yaml      # 龙虾⇔大脑通信协议契约
├── state-unit-schema.yaml  # 最小状态单元契约
└── artifact-schema.yaml    # 档案交接格式契约
```

**规则：**
- 契约由 Infra 代理初始化，Architect 审批
- 任何板块代理可以提议修改契约
- 契约变更必须经 Architect 审批 + 受影响板块代理确认
- 契约一旦发布，实现必须向后兼容（除非 Architect 批准破坏性变更）

### 3.2 依赖通知机制

当一个代理完成了另一个代理依赖的产出物时，主动通知：

```
Infra 完成 Neo4j Schema
  → 通知 Brain: "你依赖的 Schema 就绪，可以开始开发"
  → 通知 Architect: "Infra Schema 已交付"

Brain 完成意图图谱 API
  → 通知 Workflow: "意图图谱 API 就绪"
  → 通知 Frontend: "大脑 API 就绪"
  → 通知 Architect: "Brain API 已交付"
```

### 3.3 接口对齐会议（Architect 召集）

在以下时机，Architect 召集相关代理对齐：

| 时机 | 参与者 | 目的 |
|------|--------|------|
| 项目启动 | 全部 | 确认契约初版、确认依赖顺序 |
| Infra 交付后 | Brain + Workflow + Connector | 确认基础设施满足需求 |
| Brain API 就绪 | Workflow + Frontend | 确认接口满足下游需求 |
| 事件总线就绪 | Workflow + Frontend + Connector | 确认实时链路跑通 |
| 全板块交付 | 全部 | 端到端集成验证 |

### 3.4 冲突解决流程

```
板块A 和 板块B 对接口有分歧
  → 双方各提方案和理由
  → Architect 评估：
     ├── 能向后兼容？ → 选更灵活的方案
     ├── 不能兼容？   → 选对整体架构更优的方案
     └── 都合理？     → Architect 拍板，记录决策理由
  → 决策写入 contracts/ 目录
  → 受影响方按新契约调整
```

## 5. 代理间的关键数据流

### 5.1 意图→执行 完整链路

```
用户对话
  → [Frontend] 捕获语义，发送到 Brain API
  → [Brain] 更新意图图谱，触发 state_changed 事件
  → [Connector] 事件总线广播
  → [Workflow] 收到意图变更，重新编译工作流
  → [Workflow] 生成 Temporal Workflow，下发任务
  → [Connector] 通过适配器将任务发给龙虾
  → [Connector] 龙虾完成后，Output Hook 捕获结果
  → [Brain] 状态回写，认知判断
  → [Connector] 事件总线广播状态变更
  → [Frontend] WebSocket 收到事件，地图节点更新
```

### 5.2 认知闭环链路

```
  → [Connector] 龙虾执行数据积累
  → [Brain] 偏差超出容忍范围 → 触发认知节点
  → [Brain] AI 写入认知，通知相关人
  → [Connector] 事件总线广播 cognitive_trigger
  → [Workflow] 认知反向影响工作流约束
  → [Frontend] 认知层可视化更新
```

### 5.3 档案交接链路

```
  → [Workflow] 节点A完成，产出成果
  → [Connector] 成果上传到档案库，触发 artifact_uploaded
  → [Connector] 事件总线广播
  → [Workflow] 下游节点B收到事件，拉取档案内容
  → [Brain] 档案元数据写入，关联任务链
```

## 6. Architect 的监督清单

### 每日检查

- [ ] 各代理进度是否符合阶段计划
- [ ] 是否有阻塞的依赖未解决
- [ ] 契约是否有待审批的变更

### 里程碑检查

- [ ] **M1（Infra 交付）**：最小状态单元 Schema 通过全员评审
- [ ] **M2（Brain API 就绪）**：意图图谱 CRUD + 权限推导可用
- [ ] **M3（事件总线就绪）**：端到端事件收发验证通过
- [ ] **M4（工作流基本链路）**：意图 → Temporal → 执行 → 状态回写
- [ ] **M5（前端基本链路）**：对话 → 地图实时响应
- [ ] **M6（集成验证）**：完整链路端到端跑通

### 一致性检查

- [ ] 所有板块使用的事件结构是否与 `event-schema.yaml` 一致
- [ ] Brain API 的实际响应是否与 `brain-api.yaml` 契约一致
- [ ] 龙虾通信协议实现是否与 `protocol-spec.yaml` 一致
- [ ] 前端消费的数据格式是否与后端实际输出一致

## 7. 代理启动指令模板

每个板块代理启动时，应获得以下上下文：

```
1. 本板块需求文档（docs/0X_板块名.md）
2. 完整需求文档（ClawTeams_需求文档_v2.1.md）— 只读参考
3. 共享契约目录（contracts/）— 必须遵守
4. 当前已就绪的上游产出物
5. 下游板块对本板块的接口期望
```

### Architect 启动指令

```
你是 ClawTeams 的架构师代理。你不写代码，只做以下工作：
1. 审查各板块代理提交的接口变更
2. 仲裁板块间的接口分歧
3. 验证各板块实现与需求文档的一致性
4. 管理依赖和启动顺序
5. 在里程碑节点做集成验证

你的权威来源是 docs/ 目录下的需求文档和 contracts/ 目录下的接口契约。
任何偏离需求文档的实现都需要你的审批。
```

### 板块代理启动指令（以 Brain 为例）

```
你是 ClawTeams 的大脑板块代理。你的职责范围是 docs/02_团队大脑.md 中定义的所有功能。

你必须：
1. 严格遵守 contracts/ 目录中的接口契约
2. 完成需求文档中的交付标准
3. 在接口变更时通知 Architect 审批
4. 在依赖产出就绪时通知下游代理

你不可以：
1. 修改其他板块的代码
2. 绕过 Architect 直接变更契约
3. 引入契约中未定义的外部依赖
```
